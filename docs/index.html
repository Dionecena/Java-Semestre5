⚙️Installation & Configuration

Guide pas-à-pas pour démarrer l'application

Prérequis

OutilVersionVérification
Java JDK17+java -version
Maven3.8+mvn -version
PostgreSQL12+psql --version

Étapes

1. Cloner le projet

git clone https://github.com/Dionecena/Java-Whatsapp.git
cd Java-Whatsapp

2. Créer la base de données PostgreSQL

-- Se connecter à PostgreSQL
psql -U postgres

-- Créer la base
CREATE DATABASE messagerie_db;

-- Vérifier
\l
ℹ️ Tables automatiques : Hibernate crée les tables users et messages automatiquement au premier démarrage (hbm2ddl.auto=update).

3. Configurer Hibernate

cp src/main/resources/hibernate.cfg.xml.example src/main/resources/hibernate.cfg.xml

Éditer hibernate.cfg.xml :

<property name="hibernate.connection.url">
    jdbc:postgresql://localhost:5432/messagerie_db
</property>
<property name="hibernate.connection.username">postgres</property>
<property name="hibernate.connection.password">VOTRE_MOT_DE_PASSE</property>

4. Compiler

mvn clean compile

5. Lancer le serveur

mvn exec:java -Dexec.mainClass="com.example.whatsapp_java.server.ServerMain"

Sortie attendue :

[2026-02-28 10:00:00] [SERVER] Démarrage sur le port 5000

6. Lancer le client

mvn javafx:run

📡Protocole Réseau

Toutes les commandes, réponses et codes d'erreur du protocole TCP texte

Format général

COMMANDE|argument1|argument2|...\n
  • Séparateur : | (pipe)
  • Terminateur de ligne : \n (géré par PrintWriter.println() et BufferedReader.readLine())
  • Encodage : UTF-8
  • Transport : TCP (connexion persistante)

Commandes Client → Serveur

REGISTER — Inscription

FormatREGISTER|username|passwordHash
Paramètresusername : nom d'utilisateur (non vide)
passwordHash : SHA-256 du mot de passe
Réponse succèsOK|REGISTER
Réponses erreurERROR|Format REGISTER invalide
ERROR|Champs vides
ERROR|Username déjà utilisé
Implémenté dansClientHandler.handleRegister()

LOGIN — Connexion

FormatLOGIN|username|passwordHash
Paramètresusername : nom d'utilisateur
passwordHash : SHA-256 du mot de passe
Réponse succèsOK|LOGIN suivi de ALL_USERS|... et des messages en attente
Réponses erreurERROR|Format LOGIN invalide
ERROR|Utilisateur introuvable
ERROR|Mot de passe incorrect
ERROR|Utilisateur déjà connecté
Implémenté dansClientHandler.handleLogin()

LOGOUT — Déconnexion

FormatLOGOUT|
RéponseOK|LOGOUT puis fermeture de la connexion
Implémenté dansClientHandler.handleLogout()

USERS — Liste des utilisateurs en ligne

FormatUSERS|
RéponseUSERS|alice,bob,carol (CSV des usernames en ligne)
NoteRetourne seulement les utilisateurs ONLINE. Préférer ALL_USERS pour voir tout le monde.
Implémenté dansClientHandler.handleLine() → server.onlineUsersCsv()

ALL_USERS — Tous les utilisateurs

FormatALL_USERS|
RéponseALL_USERS|alice:ONLINE,bob:OFFLINE,carol:ONLINE
NoteRetourne TOUS les utilisateurs avec leur statut. Nécessaire pour RG6 (écrire à quelqu'un d'offline).
Implémenté dansClientHandler.handleLine() → server.allUsersCsv()

SEND — Envoyer un message

FormatSEND|destinataire|contenu
Paramètresdestinataire : username du destinataire
contenu : texte du message (1–1000 chars)
Réponse succèsOK|SEND
Réponses erreurERROR|Non authentifié
ERROR|Format SEND invalide
ERROR|Message vide
ERROR|Message trop long (max 1000 caractères)
ERROR|Destinataire introuvable
Effet secondaireSi le destinataire est en ligne : MSG|sender|content|timestamp lui est envoyé
Implémenté dansClientHandler.handleSend()

HISTORY — Historique de conversation

FormatHISTORY|autreUtilisateur
RéponseSéquence : HISTORY_BEGIN| → N × HISTORY|from|content|timestamp → HISTORY_END|
Réponses erreurERROR|Non authentifié
ERROR|Format HISTORY invalide
ERROR|Utilisateur introuvable
Implémenté dansClientHandler.handleHistory()

-- Table users
CREATE TABLE users (
    id             BIGSERIAL    PRIMARY KEY,
    username       VARCHAR(255) NOT NULL UNIQUE,  -- RG1: unicité
    password       VARCHAR(255) NOT NULL,          -- RG9: hash SHA-256 (64 chars hex)
    status         VARCHAR(10)  NOT NULL           -- RG4: 'ONLINE' ou 'OFFLINE'
                   CHECK (status IN ('ONLINE', 'OFFLINE')),
    date_creation  TIMESTAMP    NOT NULL
);

-- Index automatique sur username (contrainte UNIQUE)

-- Table messages
CREATE TABLE messages (
    id           BIGSERIAL     PRIMARY KEY,
    sender_id    BIGINT        NOT NULL REFERENCES users(id),
    receiver_id  BIGINT        NOT NULL REFERENCES users(id),
    contenu      VARCHAR(1000) NOT NULL,  -- RG7: max 1000 chars
    date_envoi   TIMESTAMP     NOT NULL,  -- RG8: pour tri chronologique

);

Requêtes importantes

Récupérer l'historique entre deux utilisateurs (RG8)

SELECT m.*, u1.username AS sender_name, u2.username AS receiver_name
FROM messages m
JOIN users u1 ON m.sender_id = u1.id
JOIN users u2 ON m.receiver_id = u2.id
WHERE (m.sender_id = :u1_id AND m.receiver_id = :u2_id)
   OR (m.sender_id = :u2_id AND m.receiver_id = :u1_id)
ORDER BY m.date_envoi ASC;

Récupérer les messages non lus (RG6)

SELECT m.*, u.username AS sender_name
FROM messages m
JOIN users u ON m.sender_id = u.id
WHERE m.receiver_id = :user_id
  AND m.statut != 'LU'
ORDER BY m.date_envoi ASC;

Mettre à jour le statut d'un utilisateur

UPDATE users SET status = 'ONLINE' WHERE username = 'alice';
UPDATE users SET status = 'OFFLINE' WHERE username = 'alice';

Cycle de vie d'un message

INSERT INTO messages (..., statut='ENVOYE') │ ├── Destinataire en ligne │ └── UPDATE messages SET statut='RECU' WHERE id=? │ └── Destinataire hors ligne └── (reste ENVOYE) │ └── Destinataire se reconnecte └── UPDATE messages SET statut='LU' WHERE id=?

@Entity                    // Marque la classe comme entité JPA
@Table(name = "users")     // Nom de la table en DB
@Data                      // Lombok: génère getters, setters, equals, hashCode, toString
@NoArgsConstructor         // Lombok: constructeur sans args (requis par JPA)
@AllArgsConstructor        // Lombok: constructeur avec tous les args
public class User {

    @Id                                              // Clé primaire
    @GeneratedValue(strategy = GenerationType.IDENTITY) // Auto-incrémenté (BIGSERIAL)
    private Long id;

    @NotBlank                                        // Validation: non vide
    @Column(unique = true, nullable = false)         // Contrainte DB: UNIQUE NOT NULL
    private String username;

    @NotBlank
    @Column(nullable = false)
    private String password;                         // Hash SHA-256 (64 chars hex)

    @NotNull
    @Enumerated(EnumType.STRING)                     // Stocké comme 'ONLINE'/'OFFLINE'
    @Column(nullable = false)
    private UserStatus status;

    @NotNull
    @Column(nullable = false)

}

Message.java

Package : com.example.whatsapp_java.entities
Table DB : messages

@Entity
@Table(name = "messages")
@Data
@NoArgsConstructor
@AllArgsConstructor
public class Message {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @NotNull
    @ManyToOne(fetch = FetchType.EAGER)              // Relation N:1 avec User
    @JoinColumn(name = "sender_id", nullable = false) // Colonne FK en DB
    private User sender;

    @NotNull
    @ManyToOne(fetch = FetchType.EAGER)
    @JoinColumn(name = "receiver_id", nullable = false)
    private User receiver;

    @NotBlank
    @Size(max = 1000)                                // Validation: max 1000 chars
    @Column(length = 1000, nullable = false)
    private String contenu;

    @NotNull
    @Column(nullable = false)
    private LocalDateTime dateEnvoi;

    @NotNull
    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    private MessageStatus statut;
}

@ManyToOne expliqué

@ManyToOne signifie "plusieurs messages peuvent avoir le même expéditeur". Hibernate génère une colonne sender_id (clé étrangère) qui référence users.id.

FetchType.EAGER : Hibernate charge l'objet User associé immédiatement lors du chargement du Message (pas de lazy loading).

RG1Username unique✅ Implémenté

Le username doit être unique.

Où : ClientHandler.handleRegister() + contrainte @Column(unique=true) sur User.username

RG3Un seul login à la fois✅ Implémenté

Un utilisateur ne peut être connecté qu'une seule fois simultanément.

Où : ChatServer.registerActive() utilise ConcurrentHashMap.putIfAbsent()

// ChatServer.java
public boolean registerActive(String username, ClientHandler handler) {
    // putIfAbsent est atomique : retourne null si la clé n'existait pas
    return activeUsers.putIfAbsent(username, handler) == null;
}
// ClientHandler.java – handleLogin()
if (server.isUserOnline(u)) {
    sendLine("ERROR|Utilisateur déjà connecté"); return;
}
boolean ok = server.registerActive(u, this);
if (!ok) {
    sendLine("ERROR|Utilisateur déjà connecté"); return;
}

Pourquoi deux vérifications ? La première (isUserOnline) est rapide. La seconde (registerActive avec putIfAbsent) est atomique et protège contre les race conditions.

RG4Statut ONLINE/OFFLINE✅ Implémenté

À la connexion → ONLINE. À la déconnexion ou perte réseau → OFFLINE.

Où : ClientHandler.handleLogin() et ClientHandler.cleanup()

// À la connexion
userDAO.updateStatus(u, UserStatus.ONLINE);

// cleanup() est dans le bloc finally → exécuté TOUJOURS
// (déconnexion volontaire OU perte réseau)
private void cleanup() {
    if (username != null) {
        userDAO.updateStatus(username, UserStatus.OFFLINE);
        server.unregister(username);
        broadcastAllUsers();
        username = null;
    }
    socket.close();
}

RG6Messages hors ligne livrés à la reconnexion✅ Implémenté

Si le destinataire est hors ligne, le message est enregistré et livré lors de sa prochaine connexion.

Où : ClientHandler.handleSend() (sauvegarde) + ClientHandler.handleLogin() (livraison)

// handleLogin() : livraison des messages en attente

    messageDAO.updateStatus(m.getId(), MessageStatus.LU);
}

Établit la connexion TCP et démarre le thread de lecture. Le thread est un daemon thread (s'arrête avec l'application). Appelle listener.onLineReceived() pour chaque ligne reçue, et listener.onDisconnected() en cas de fermeture.
public void connect(Listener listener) throws IOException {
    this.listener = listener;
    socket = new Socket(host, port);
    in  = new BufferedReader(new InputStreamReader(socket.getInputStream()));
    out = new PrintWriter(new OutputStreamWriter(socket.getOutputStream()), true);

    readerThread = new Thread(() -> {
        try {
            String line;
            while ((line = in.readLine()) != null) {
                Listener l = this.listener;
                if (l != null) l.onLineReceived(line);
            }
            Listener l = this.listener;
            if (l != null) l.onDisconnected(null);
        } catch (Exception e) {
            Listener l = this.listener;
            if (l != null) l.onDisconnected(e);
        }
    });
    readerThread.setDaemon(true);
    readerThread.start();
}

public class HibernateUtil {
    private static SessionFactory sessionFactory;

    public static SessionFactory getSessionFactory() {
        if (sessionFactory == null) {
            synchronized (HibernateUtil.class) {
                if (sessionFactory == null) {
                    try {
                        URL cfg = HibernateUtil.class.getClassLoader()
                                      .getResource("hibernate.cfg.xml");
                        System.out.println("[HIBERNATE] Config: " + cfg);
                        sessionFactory = new Configuration()
                                            .configure("hibernate.cfg.xml")
                                            .buildSessionFactory();
                    } catch (Throwable ex) {
                        System.err.println("SessionFactory failed: " + ex.getMessage());
                        throw ex;
                    }
                }
            }
        }
        return sessionFactory;
    }

    public static void shutdown() {
        try {
            if (sessionFactory != null) {
                sessionFactory.close();
                sessionFactory = null;
            }
        } catch (Exception ignored) {}
    }
}

Méthodes

public static SessionFactory getSessionFactory()
SessionFactory — instance unique UserDAO (toutes méthodes), MessageDAO (toutes méthodes) ExceptionInInitializerError si hibernate.cfg.xml manquant ou DB inaccessible
Retourne la SessionFactory unique. Utilise le pattern double-checked locking pour garantir la thread-safety et l'initialisation unique. La SessionFactory est créée en lisant hibernate.cfg.xml depuis le classpath.
public static void shutdown()
Non appelé dans le code actuel (à appeler à l'arrêt de l'application)

Double-checked locking expliqué

if (sessionFactory == null) {          // 1ère vérification (rapide, sans lock)
    synchronized (HibernateUtil.class) { // Lock uniquement si null
        if (sessionFactory == null) {    // 2ème vérification (sous lock)
            sessionFactory = new Configuration().configure().buildSessionFactory();
        }
    }
}

Sans le double-check, deux threads pourraient créer deux SessionFactory simultanément. Le synchronized seul serait trop lent (lock à chaque appel).

public class PasswordUtil {
    public static String sha256(String input) {
        try {
            MessageDigest digest = MessageDigest.getInstance("SHA-256");
            byte[] hash = digest.digest(input.getBytes(StandardCharsets.UTF_8));
            StringBuilder hexString = new StringBuilder();
            for (byte b : hash) {
                String hex = Integer.toHexString(0xff & b);
                if (hex.length() == 1) hexString.append('0'); // Padding zéro
                hexString.append(hex);
            }
            return hexString.toString(); // 64 caractères hexadécimaux
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
}

Platform.runLater()

Platform.runLater(() -> handleServerLine(line));

Problème : JavaFX n'est pas thread-safe. L'interface graphique ne peut être modifiée que depuis le JavaFX Application Thread. Le thread de lecture du ChatClient est un thread séparé.

⚠️ Règle : Toute modification de l'UI (setText, getItems().add, etc.) depuis un thread non-JavaFX DOIT passer par Platform.runLater().

Explications des concepts Hibernate utilisés dans le projet

SessionFactory

La SessionFactory est l'objet central d'Hibernate. Elle est créée une seule fois au démarrage (coûteux) et réutilisée pour créer des Session.

// Création (une seule fois)
SessionFactory sf = new Configuration().configure("hibernate.cfg.xml").buildSessionFactory();
// Utilisation
try (Session session = sf.openSession()) { ... }

Session

Une Session représente une connexion à la DB. Elle est légère et doit être ouverte/fermée pour chaque opération (pattern try-with-resources).

try (Session session = HibernateUtil.getSessionFactory().openSession()) {
    // La session est automatiquement fermée à la fin du bloc
    User user = session.get(User.class, 1L);
}

Transaction

Les opérations d'écriture (INSERT, UPDATE, DELETE) nécessitent une transaction explicite.

Transaction tx = null;
try (Session session = sf.openSession()) {
    tx = session.beginTransaction();
    session.persist(entity);  // INSERT
    tx.commit();              // Valider
} catch (Exception e) {
    if (tx != null) tx.rollback(); // Annuler en cas d'erreur
    throw e;
}

HQL (Hibernate Query Language)

HQL est similaire à SQL mais utilise les noms de classes et d'attributs Java (pas les noms de tables/colonnes).

// HQL (utilise le nom de classe Java "User", pas "users")
"from User u where u.username = :username"
// SQL généré : SELECT * FROM users WHERE username = ?

// HQL avec jointure implicite
"from Message m where m.sender = :user"
// SQL généré : SELECT m.* FROM messages m WHERE m.sender_id = ?
